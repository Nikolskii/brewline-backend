## Назначение репозитория
Backend Brewline: **REST API + real-time поток статусов заказов**. Центральный сервис — держит цифровую
очередь заказов и рассылает изменения статуса во все точки отображения (клиент, бариста, экран).

**⭐ Ключевая роль в архитектуре — источник истины контракта.** Форма данных задана zod-схемами
в `src/contract/schemas.ts`; из них генерируется `openapi/openapi.yaml`, а из него — npm-пакет
**`@brewline/api-types`**, которым питаются фронты (`brewline-display` и др.).
Это значит: меняешь контракт — думаешь о потребителях. (ADR 0008, 0009, 0010.)

**Цепочка:** zod-схемы → `openapi.yaml` → `@brewline/api-types` → фронты.
`openapi.yaml` **генерируется и коммитится** — править его руками бессмысленно, правки затрёт
`npm run gen:spec`. CI проверяет (`npm run check:spec`), что закоммиченная спека не отстала от схем.

**Definition of Done при изменении API:**
1. Правишь схему в `src/contract/schemas.ts` (и путь в `scripts/generate-openapi.ts`, если меняются эндпоинты).
2. `npm run gen:spec` — перегенерировать спеку, закоммитить её вместе с кодом.
3. Поднять версию в `packages/api-types/package.json` (SemVer: несовместимое изменение = major).
4. Запушить тег `api-types-v<версия>` — workflow опубликует пакет.

Без шага 4 фронты не увидят изменения.

**Порядок работы (важно, см. цену в ADR 0010):** схемы пишутся **до** реализации логики. Спека
теперь производная от кода, и если сначала писать логику, фронты окажутся заблокированы ожиданием
backend — ровно то, чего мы избегали, когда спека была рукописной.

## Стек
- **Node.js + Express + TypeScript** (TS strict).
- **MongoDB** — хранилище заказов (официальный драйвер `mongodb`). ADR 0002 (Mongo под модель заказов — открытый вопрос, помечать сомнения).
- **Real-time — SSE (Server-Sent Events)**, не WebSocket. Модель: «снапшот» — сервер шлёт актуальное
  состояние очереди, payload переиспользует доменный тип `Order` из той же OpenAPI (отдельного языка событий нет). ADR 0001.

## Границы инкремента 1 (walking skeleton)
Скоуп именно этого среза (декомпозиция — в vault, `planning/Декомпозиция — инкремент 1 …`):
- Модель `Order` в MongoDB + **автомат статусов** (I3).
- **OpenAPI-спека** REST + доменные типы (I4) — делаем в первую очередь, разблокирует фронт.
- `GET /orders` — снапшот очереди.
- Смена статуса заказа (эндпоинт).
- `GET /orders/stream` — SSE-поток снапшотов.
Вне скоупа среза 1: создание заказов из кассы/веб-формы, аутентификация, деплой (это срез 2).

## Как связан с соседями
- `brewline-display` потребляет: `GET /orders` (снапшот) + `GET /orders/stream` (SSE), а типы берёт
  из нашего пакета `@brewline/api-types` (собственной копии спеки у фронтов больше нет).
- `brewline-infra` — общекластерное/прод (в т.ч. прод-Mongo кластера, срез 2). Dev-Mongo — здесь (см. ниже).

## Локальный запуск
- **dev-стек живёт в этом репо**: `docker compose up` поднимает Mongo + backend (dev, hot-reload). `docker-compose.yml` — рядом. Репо владеет своим dev-окружением, без связи с путями соседних репо.
- Конфиг — через переменные окружения (`MONGO_URL`, `PORT`). Реальные значения — в `.env` (не коммитить), пример — в `.env.example`.

## Безопасность (репо публичный)
- **Никогда не коммитить секреты**: `.env`, `MONGO_URL` с паролем, токены. Только плейсхолдеры и `.env.example`.

## Конвенции
- TypeScript strict; явные доменные типы.
- ADR пишем, когда упёрлись в развилку (скилл `adr`), а не заранее.
